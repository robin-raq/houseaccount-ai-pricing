Documented modeling approach

How the estimate is made

A transparent model card: the data, the features, the model, and — most importantly — how confidence is computed and when the engine admits it's guessing.

Data used

  • 1,432 sanitized historical jobs across 18 service categories
  • 411 rows carry a final_price — the supervised training/eval signal
  • 1,033 unique real ZIP codes, joined to median household income (Census)
  • PII stripped; dates truncated to YYYY-MM; 13 jobs > $10K dropped as outliers

Features

  • Category & subtype (normalized title-case ↔ kebab-case)
  • ZIP → region + median household income (demographic signal, not a price table)
  • Booking month → seasonality; deadline → urgency
  • LLM scope signals from the description: size, fixture count, complexity, material tier, est. labor hours, who-supplies-parts

Model

  • Gradient-boosted regressor on log-price (robust to the skewed price distribution)
  • Conformal prediction for the lo / hi interval — calibrated, distribution-free
  • Deterministic core: same input → same estimate, every time

How confidence is calculated

Base confidence comes from the conformal interval width relative to the point estimate — a tight band means the model is sure, a wide band means it isn't. It's mapped monotonically into [0, 1] and clamped.

Out-of-distribution → forced < 0.5

These never get rejected or capped — they pass through, flagged, so the marketplace can route them:

  • Estimate midpoint > $5,000 (≈95th percentile of training)
  • Interval wider than 3× the median observed range
  • Category outside the 10 current production verticals

Known limitations

  • No square-footage / fixture fields in source — scope is inferred from text
  • Sparse categories were augmented; real-job confidence is lower there
  • Month-level dates only — no day-of-week or precise seasonality
The contract

API & booking-flow integration

The Rails service implements the Appendix A contract: bearer auth, structured booking in, structured estimate out, and an outbound post into the HouseAccount booking flow.

Endpoint

POST  /pricing-estimate

Header Authorization: Bearer <secret> · constant-time compared · content-type application/json.

Request

// required: job_id, service_category, zip_code, job_description
{
  "job_id": "abc123",
  "service_category": "Plumbing",
  "service_subtype": "Water Heater Replacement",
  "zip_code": "78704",
  "job_description": "50-gal gas water heater, pilot won't stay lit.",
  "deadline": "Within 1-2 weeks",
  "booking_month": "2026-05"
}

Response · 200

{
  "ok": true,
  "job_id": "abc123",
  "estimate_lo": 1450,
  "estimate_hi": 2200,
  "estimate_midpoint": 1825,
  "confidence": 0.78,
  "model_version": "houseaccount-pricing-v1.0.0"
}

Try it

curl -X POST $URL/pricing-estimate \
  -H "Authorization: Bearer $SECRET" \
  -H "Content-Type: application/json" \
  -d '{ "job_id":"demo","service_category":"Plumbing",
       "zip_code":"78704",
       "job_description":"Replace 50-gal water heater" }'
200 OK · 0.74 s 
{ "ok":true, "estimate_midpoint":1825, "confidence":0.78 }
